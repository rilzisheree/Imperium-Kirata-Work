Before making any changes, thoroughly read and understand my existing admin command system, camera system, RemoteEvents, player handling, and coding style. Reuse the current architecture and only add the functionality described below.

Create two new admin commands:

watch <target>

unwatch

--------------------------------------------------
WATCH
--------------------------------------------------

Description:
- Makes the admin view the specified player's point of view.

Requirements:
- Follow the existing admin command framework and coding style.
- Support all existing target selectors (username, display name, me, etc.) if the command system already provides them.
- When executed, smoothly transition the admin's camera to spectate the target player.
- The camera should follow the target player exactly as if the admin were playing as them.
- Preserve the target player's normal gameplay; they should not notice they are being watched.
- If the target respawns, automatically continue spectating their new character.
- If the target leaves the game, immediately stop spectating, restore the admin's normal camera, and display a small notification explaining that the player left.
- Only the admin who executed the command should have their camera changed.
- Prevent duplicate spectating sessions.

While Spectating:
- Display a small, clean indicator at the top-center of the screen:
  **Watching: <Display Name> (@Username)**
- Keep the indicator visible until spectating ends.

--------------------------------------------------
UNWATCH
--------------------------------------------------

Description:
- Stops spectating and returns the admin's camera to normal.

Requirements:
- Restore the admin's original CameraType, CameraSubject, FieldOfView, and any other camera properties that were modified.
- Remove the "Watching" indicator.
- If the admin is not currently spectating anyone, do nothing without throwing errors.
- Cleanly disconnect any events or camera update loops used by the spectate system.
- Handle cases where the target has already left or respawned.

--------------------------------------------------
GENERAL REQUIREMENTS
--------------------------------------------------

- Executing `watch` while already spectating another player should automatically switch to the new target.
- If the admin dies or resets while spectating, automatically restore their normal camera.
- Keep camera movement smooth without jittering or desynchronization.
- Prevent memory leaks and duplicate connections.
- Integrate seamlessly into the existing admin system without breaking any current functionality.
- Follow the existing coding style and project architecture.
